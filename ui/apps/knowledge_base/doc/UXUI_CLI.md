# Концепты UX/UI на базе CLI-инструментов

В данном документе собраны основные паттерны и концепты отображения данных, которые уже реализованы в консольных утилитах (`inspect_graph.py`, `inspect_tree.py`, `inspector.py`), для их последующего переноса в полноценный графический интерфейс (Knowledge Base UI).

---

## 1. Паттерны отображения графа (на базе `inspect_graph.py`)

### 1.1. Табличные списки (List Views)
- **Глобальный справочник сущностей (Keywords)**:
  - Таблица всех уникальных сущностей базы знаний.
  - Колонки: Ключевое слово, Категория, Максимальная уверенность (Max Conf), Количество статей, Из каких слов было "склеено" (Merged from).
  - Полезно для дашборда или раздела "Словарь/Глоссарий".
- **Срез по конкретной статье (Article View)**:
  - Таблица сущностей, найденных только в конкретной статье.
  - Колонки дополняются контекстом статьи: Уверенность в этой статье, Количество чанков (упоминаний), Оригинальные формы слов.
  - **UX-элемент**: "Распределение уверенности" (Confidence distribution) — индикатор качества извлечения для статьи (сколько high (≥0.8), med (0.5–0.8), low (<0.5) уверенности).

### 1.2. Карточка сущности (Keyword Detail View)
В CLI реализовано в виде древовидной структуры. В UI это может быть "Карточка/Профиль сущности" с вкладками:
- **Заголовок**: Название сущности, Категория, Из каких оригинальных слов слито.
- **Вкладка "Связи" (Relations)**:
  - Список фактов вида: `Субъект --[Предикат]--> Объект`.
  - **Ключевой UX-паттерн ("Traceability / Доказуемость")**: Связи сгруппированы по ID чанка (текстового фрагмента). Рядом со связью всегда можно развернуть/увидеть **исходный текст** из Qdrant, из которого эта связь была извлечена.
- **Вкладка "Статьи" (Articles)**:
  - Список статей, в которых упоминается сущность. Для каждой статьи показывается уверенность, количество чанков упоминия и локальное описание (context description).
- **Вкладка "Кросс-ссылки" (Cross-References)**:
  - Если сущность является названием другой статьи (ссылкой), показываются места (с исходным текстом чанка), где одна статья ссылается на другую с указанием анкора (display) и раздела (section).

## 2. Паттерны отображения дерева (на базе `inspect_tree.py`)

### 2.1. Иерархическое чтение (RAPTOR Tree View)
Визуализация того, как алгоритм RAPTOR сжимает статью по уровням семантики.
- **UX-представление**: Многоуровневое дерево (Accordion, TreeList или визуальный граф).
- **Цветовое и символьное кодирование**:
  - `📄 Level 0 (Листья)`: Оригинальные куски текста (исходный текст).
  - `📝 Level 1+ (Родители/Саммари)`: Сгенерированные LLM обобщения дочерних кусков текста.
- **Превью текста**: Карточки узлов показывают первые 150 символов с возможностью развернуть ("Читать далее") полный текст.
- **Саммари документа**: Отдельно, на самом верху, выводится глобальное "Саммари статьи" (хранимое в Neo4j), которое является смысловой выжимкой из корневых узлов дерева.

## 3. Паттерны отображения концептов (на базе `inspector.py`)

В модуле `concept_builder` реализована логика "отслеживания происхождения" (Provenance Tracing) от макро-концепта к микро-упоминаниям.

### 3.1. Глубокая инспекция концепта (Concept Inspector)
- **Шапка концепта**:
  - Каноническое имя (Canonical Name), Домен/Предметная область, Обобщенное описание (Description).
- **Ключевой UX-паттерн ("Ранжированные доказательства" / Ranked Evidence)**:
  - Концепт состоит из нескольких "Keyword" (сущностей-синонимов из разных статей).
  - UI должен отображать список этих сущностей, **отсортированных по семантической близости** (similarity) к абстрактному описанию концепта (насколько конкретное слово в статье соответствует общему концепту).
- **Матрешка Traceability (Drill-down)**:
  Сценарий погружения для пользователя:
  1. Пользователь читает описание абстрактного **Концепта**.
  2. Разворачивает список **Статей**, формирующих концепт.
  3. Внутри статьи видит конкретное **Ключевое слово** (сущность) и локальное описание.
  4. Раскрывает сущность и видит набор **Исходных чанков текста** (цитаты из `raptor_chunks`), откуда ഇത് было взято.
- **Связи между концептами (Cross-relations)**:
  - Отображение связей уровня концепт-концепт (`CROSS_RELATED_TO`) с описанием (Description) и уверенностью.

## 4. Паттерны управления жизненным циклом концептов (Versioning & Creation)

В модуле `concept_builder` реализованы механизмы инкрементального обновления и ручного создания концептов, которые должны быть отражены в графическом интерфейсе.

### 4.1. Обновление и версионирование концептов (Expand)
Концепты не статичны. При появлении новых статей база знаний может предложить расширить существующие концепты (команда `expand`).
- **Сценарий UI ("Предпросмотр обновления")**:
  - Пользователь выбирает концепт и "накидывает" в него новые статьи.
  - Система сравнивает векторы новых сущностей (Keywords) из этих статей с вектором Концепта.
  - **Интерфейс "Сравнение версий (Diff View)"**:
    - `v1 (Текущая)`: Старый набор слов и описание.
    - `v2 (Прямые совпадения)`: Добавляются слова с высокой семантической близостью (cosine similarity ≥ `high_threshold`).
    - `v3 (LLM-проверенные)`: Добавляются слова со средней близостью, которые дополнительно прошли проверку через LLM.
  - Пользователь выбирает, какую версию принять.
  - Старая версия остается в базе (`is_active = False`), к новой протягивается связь `[:EVOLVED_TO]`. В UI полезно иметь "Историю версий" (History Log) для каждого концепта.

### 4.2. Ручное создание концептов (Manual Concept Creation)
Система позволяет не только автоматически кластеризовать сущности, но и создавать знания "сверху вниз" (команда `add-concept`).
- **Сценарий UI ("Мастер создания концепта")**:
  1. Пользователь вводит: Название, Домен и базовое Описание (Description).
  2. Пользователь выбирает набор статей (Article Pool), в которых система должна искать доказательства.
  3. Система ищет подходящие сущности в этих статьях на основе векторной близости к введенному описанию.
  4. **Интерфейс "Ревью кандидатов"**:
     - Блок 1: Точные совпадения (Direct matches) — добавляются автоматически.
     - Блок 2: Кандидаты (Candidates) — требуют ручного подтверждения галочками.
  5. После объединения LLM переписывает/обогащает исходное описание пользователя на основе найденных фактов. Пользователь получает готовый узел Concept (`is_manual=True`) со всеми связями `INSTANCE_OF` к статьям.

### 4.3. Управление набором статей (Article Pool Management)
- В UI интерфейса концепта должен быть удобный компонент (например, список тегов-ID статей с кнопкой "Добавить статьи" или "Удалить"). 
- Привязка новых статей запускает процесс `Expand` (поиск новых сущностей и пересчет описания с версионированием).

### 4.4. Отслеживание версий исходных текстов (Source Text Snapshots)
В интерфейсе необходимо заложить возможность просмотра состояния исходного текста статьи **на момент формирования концепта** (поле `source_versions` у узла Concept).
- **Сценарий UI ("Snapshot View / Outdated Alert")**:
  - Если статья была обновлена с момента создания концепта, UI концепта должен подсвечивать этот факт (маркер "Outdated Source").
  - В панелях "Traceability" (где показывается исходный текст) должна быть возможность переключаться между:
    1. **Snapshot**: Текст чанка, каким он был при сборке концепта.
    2. **Current**: Актуальный текст чанка в базе (с подсветкой изменений / diff, если возможно).
  - Наличие обновленной статьи должно предлагать пользователю кнопку "Пересчитать / Обновить концепт", запускающую перегенерацию связей для новой версии статьи с созданием новой версии концепта.

## Итоговые выводы для UI/UX

1. **Доказуемость (Traceability) как основа UX.** Ни один сгенерированный факт/концепт не должен "висеть в воздухе". UI должен предоставлять бесшовный переход (drill-down) от высокоуровневых саммари и концептов вплоть до подсвеченных абзацев в исходных статьях.
2. **Панели исходного текста.** Повсеместное использование карточек/панелей (как в Rich CLI) для отображения `chunk text` рядом с извлеченными фактами или связями.
3. **Разделение режимов списка и иерархии.** Дать возможность пользователю читать статьи двояко: линейно (или иерархически по RAPTOR) и через граф сущностей/концептов.
4. **Интерактивное курирование знаний (Human-in-the-loop).** UI должен поддерживать процессы ревью кандидатов (при создании/расширении) и выбор версий (v1 vs v2) вместо автоматической перезаписи данных.
