# Полный список функций UI-платформы

Документ объединяет возможности React Flow (`@xyflow/react` v12) с требованиями из `STRUCTURE.md`, `UXUI_CLI.md`, `UXUI_DAG.md` и `TECH_REQUIREMENTS.md`.  
Каждая функция отвечает на вопрос **«зачем»** — какую задачу пользователя она решает.

---

# 🔧 DAG Builder (Оркестратор пайплайнов)

## A. Canvas — визуальный редактор графа

| # | Функция | Зачем | React Flow фича |
|---|---|---|---|
| A1 | **Drag-and-drop добавление нод** из каталога шагов | Чтобы собирать пайплайн мышкой, не правя YAML руками | `onDrop` + `screenToFlowPosition` (пример "Drag and Drop") |
| A2 | **Автолейаут графа** (Dagre / Elkjs) | Чтобы при загрузке YAML или добавлении нод граф красиво располагался автоматически, без ручного выравнивания | Примеры "Dagre Layout" и "Elkjs Layout" |
| A3 | **MiniMap** (навигационная миникарта) | Чтобы ориентироваться в больших пайплайнах (10+ шагов) без потери контекста | Встроенный `<MiniMap />` |
| A4 | **Controls** (зум, фит, блокировка) | Масштабирование графа, кнопка "Показать весь граф" одним кликом | Встроенный `<Controls />` |
| A5 | **Сетка / Background** | Визуальная привязка к сетке для аккуратного расположения нод | Встроенный `<Background variant="dots" />` |
| A6 | **Выравнивание по направляющим (Helper Lines)** | Чтобы ноды выравнивались друг к другу при перетаскивании, как в Figma | Пример "Helper Lines" |
| A7 | **Prevent Cycles** | Запретить создание циклических зависимостей прямо при подключении ребра | `isValidConnection` (пример "Prevent Cycles") |
| A8 | **Undo / Redo** | Откат последних действий (удалил ноду — отменить) | Пример "Undo/Redo" (стеки действий) |
| A9 | **Save / Restore** | Сохранение текущего состояния графа (localStorage / API) и восстановление | Пример "Save and Restore" |
| A10 | **Selection grouping** | Выделить несколько нод и сгруппировать в логический блок (SubFlow) | Примеры "Sub Flows" + "Selection Grouping" |
| A11 | **Add Node on Edge Drop** | При перетаскивании ребра в пустоту — предложить выбрать ноду для вставки | Пример "Add Node on Edge Drop" |

## B. Кастомные ноды (StepNode)

| # | Функция | Зачем |
|---|---|---|
| B1 | **I/O Ports (Handles)** — входные/выходные порты на ноде | Чтобы визуально видеть, какие данные нода принимает и отдает, и соединять их проводами |
| B2 | **Типизированные выходы (outputs)** c подписями `key: type` | Чтобы пользователь понимал, что конкретно экспортирует шаг, и мог привязать к нужному полю |
| B3 | **Бейдж модуля и tags** на ноде | Чтобы сразу видеть тип шага (`raptor_pipeline.run`) и его теги (`indexing`, `etl`) |
| B4 | **Context badges** (📤 Provides / 📥 Requires) | Чтобы видеть, какие неявные (code-level) зависимости есть между шагами, помимо `depends_on` |
| B5 | **Callback иконки** (⟳ retry, 🔔 alert, ✓/✗ success/failure) | Чтобы одним взглядом понять поведение шага при ошибке или успехе |
| B6 | **NodeToolbar** — контекстное меню при выделении | Быстрые действия: Удалить, Дублировать, Отключить, Запустить только этот шаг | 
| B7 | **NodeResizer** — изменение размера ноды | Чтобы развернуть ноду и показать больше информации (полный конфиг, превью) |
| B8 | **Цветовая индикация состояния** (idle / running / success / failed) | После запуска видеть прогресс прямо на нодах без переключения экранов |
| B9 | **Connection Limit** на Handle | Ограничить количество входящих соединений там, где вход принимает ровно одно значение |
| B10 | **Proximity Connect** | Автоматическое соединение нод при перетаскивании рядом — ускоряет сборку графа |
| B11 | **Node Appearance Templates** — кастомный внешний вид ноды | Задавать иконку, цвет, layout и видимые поля для каждого типа шага, сохранять как шаблон |

## C. Кастомные ребра (DataEdge)

| # | Функция | Зачем |
|---|---|---|
| C1 | **Подпись типа данных на ребре** (EdgeLabelRenderer) | Видеть, какой тип (`str`, `dict`, `DataFrame`) течет по связи |
| C2 | **Валидация типов при соединении** (`isValidConnection`) | Не дать соединить несовместимые порты (output `str` → input ожидает `int`) |
| C3 | **Floating Edges** — ребра прикрепляются к ближайшей стороне | Чтобы стрелки не перекрывались при сложных графах |
| C4 | **Маркер стрелки** (arrowhead) | Четкое отображение направления потока данных |
| C5 | **Визуальное разделение типов связей** (data wire vs dependency) | Различать ребра данных (`${{ }}` — сплошная) и ребра порядка (`depends_on` — пунктир) |

## D. Инспектор (Боковая панель)

| # | Функция | Зачем |
|---|---|---|
| D1 | **Автогенерация формы из JSON Schema** | Пользователь редактирует конфиг шага через форму, не зная YAML — всё из Pydantic-модели |
| D2 | **Source Badges** у каждого поля (🔵 default / 🟢 step / 🟡 override / 🔷 global) | Чтобы пользователь понимал откуда пришло значение и что он перекрывает |
| D3 | **Schema-aware фильтрация** — скрытие неприменимых глобальных ключей | Не путать пользователя: шагу `parser` не показывать `stores.neo4j` |
| D4 | **Hydra Defaults dropdowns** | Переключать конфигурационные пресеты (embeddings → huggingface/openai) одним кликом |
| D5 | **Callback Picker** — выбор из списка зарегистрированных | Не набирать имя руками, а выбрать из каталога + заполнить форму параметров |
| D6 | **Outputs editor** — объявление выходов шага | Добавлять/удалять output-ы (key + type) для последующего связывания |
| D7 | **Context Inspector** — просмотр полей dataclass контекста | Понимать, какие разделяемые данные (connections, flags) доступны downstream |
| D8 | **Autocomplete** в полях конфига (`${{ steps.` → подсказки) | Ускорить привязку данных, не запоминая имена шагов и ключей |

## E. YAML Panel (Monaco)

| # | Функция | Зачем |
|---|---|---|
| E1 | **Live Graph → YAML** (serialize, debounced) | Видеть итоговый YAML в реальном времени при любом изменении в графе |
| E2 | **Live YAML → Graph** (deserialize, debounced) | Редактировать YAML руками (копировать из Git) и мгновенно видеть граф |
| E3 | **Подсветка ошибок в Monaco** (markers) | Красные подчеркивания прямо в коде при ошибках валидации |
| E4 | **Syntax highlighting + autocomplete** для YAML | Удобная правка с подсказками ключей (`module`, `depends_on`, `config`) |

## F. Валидация и запуск

| # | Функция | Зачем |
|---|---|---|
| F1 | **Real-time cycle detection** | Не дать пользователю создать невалидный граф |
| F2 | **Missing dependency check** | Подсветить, если `depends_on` ссылается на несуществующий шаг |
| F3 | **Output ref validation** | Проверить, что `${{ steps.X.Y }}` ссылается на реальный шаг с объявленным output |
| F4 | **Type compatibility check** на ребрах | Не допустить `str → int` несовместимость между output и target field |
| F5 | **Config schema validation** | Проверить что конфиг шага проходит Pydantic + Hydra валидацию |
| F6 | **Context prerequisite check** | Подсветить, если шаг требует `RaptorContext`, а ни один upstream его не предоставляет |
| F7 | **Dry-run** (прогон без выполнения) | Посмотреть порядок, resolved-конфиги, готовность — без риска |
| F8 | **Global config Impact Preview** | При изменении global-ключа показать, какие шаги затронуты (schema-aware) |
| F9 | **Панель ошибок** (ValidationOverlay) | Кликабельный список всех проблем с навигацией к нужной ноде |

## G. Pipeline Lifecycle

| # | Функция | Зачем |

| G1 | **Git Diff** между версиями pipeline YAML | Сравнивать изменения перед мержем |
| G2 | **Rollback** к предыдущей версии | Откатиться к работающей конфигурации |
| G3 | **Templates / Subgraphs** — сохранение кусков графа | Переиспользовать блоки (ETL, Feature Eng) в разных пайплайнах |
| G4 | **Параметризация** (`${run_date}`, env presets) | Запускать один и тот же граф с разными входными параметрами |

# 📚 Knowledge Base (База знаний)

## H. Graph View — визуализация Neo4j

| # | Функция | Зачем | React Flow фича |
|---|---|---|---|
| H1 | **4 типа Custom Nodes** (Article, Keyword, Concept, ConceptInactive) | Чтобы визуально различать сущности разного уровня в графе знаний | Custom Nodes + node styling |
| H2 | **5 типов Custom Edges** (HAS_KEYWORD, INSTANCE_OF, CROSS_RELATED, EVOLVED_TO, REFERENCES) | Чтобы видеть природу связи (извлечение, кластеризация, эволюция, цитирование) | Custom Edges + EdgeLabelRenderer |
| H3 | **Layer Filters** — включение/выключение слоев графа | Не показывать сотни Keywords, когда нужен обзор Concepts; управлять плотностью | `<Panel>` с чекбоксами |
| H4 | **Similarity threshold slider** | Фильтровать слабые связи, оставляя только сильные (high confidence) | `<Panel>` |
| H5 | **Auto-layout** для подграфов из Neo4j | При подгрузке подграфа ноды не должны лежать в куче | Dagre / Elkjs layout |
| H6 | **Expand neighbors** — подгрузка соседей при клике | Граф слишком велик целиком → начинаем с одной ноды и раскрываем окружение | `updateNode` + API `graph/neighbors` |
| H7 | **Floating Edges** | RFlow при подключении динамически прикрепляет к ближайшей стороне | "Simple Floating Edges" |
| H8 | **MiniMap + Controls + Background** | Навигация по большому графу знаний | Встроенные компоненты |
| H9 | **Цветовое кодирование** нод по domain/type | Быстрая визуальная группировка концептов по областям (NLP, ML, DevOps) | CSS variables |
| H10 | **Масштаб Keyword-нод по score** | Чем выше confidence/similarity, тем крупнее отображается нода — визуальное ранжирование | Computed `width/height` |

## J. Detail Panel — инспектор элемента

## L. Concept Lifecycle — управление знаниями

| # | Функция | Зачем |
|---|---|---|
| L1 | **Expand Panel** — сравнение версий (v1 / v2 direct / v3 LLM) | Human-in-the-loop: пользователь выбирает, какую степень расширения принять |
| L2 | **Checkbox review** для candidate keywords | Подтвердить/отклонить каждый спорный keyword перед включением в концепт |
| L3 | **Create Concept Wizard** (3 шага: Define → Review → Confirm) | Создание знания «сверху вниз»: указать тему → найти доказательства → обогатить LLM |
| L4 | **Article Pool selector** | Выбрать набор статей для поиска доказательств / расширения концепта |
| L5 | **Inbox / Needs Review** — список Stale концептов | Пользователь видит что требует внимания (устаревшие источники) и может одним кликом запустить обновление |
| L6 | **Кнопка "Пересчитать / Обновить концепт"** | При обнаружении Outdated Source — запуск Expand с созданием новой версии |

# 🔗 Общие (Shared) функции обоих приложений

| # | Функция | Зачем |
|---|---|---|
| S1 | **Dark / Light mode** (`colorMode` React Flow + CSS vars) | Комфортная работа в любых условиях освещения |
