# Алгоритм проверки прав ACL (Rule Engine)

В основе `vault_acl` лежит движок `RuleResolver` (`engine.py`). Его задача — для конкретного пользователя, пути файла и запрашиваемого бита доступа определить итоговый результат (True / False).

Система позволяет создавать сложные исключения и пересечения (например, пользователю по роли разрешены все файлы `**/*`, но конкретный файл для него запрещён директивой `deny: true`, либо наоборот — на весь каталог стоит глобальный запрет, но конкретный файл персонально разрешён пользователю).

## Этапы выполнения запроса к Resolver

### 1. Нормализация пути и поиск применимых правил

При вызове `resolve_permissions(user, file_path)` алгоритм:
1. Привязывает к `user` его список ролей из реестра (Policy).
2. Проходит по всем правилам `AccessRule` из списка.
3. Проверяет паттерн `fnmatch.fnmatchcase` (пути нормализуются с использованием `/` разделителей).
4. Проверяет, подходит ли правило текущему пользователю:
   - Если имя пользователя указано в `users: [...]` у правила.
   - Если у пользователя есть хотя бы одна роль, указанная в `roles: [...]` правила.
5. Правила, не прошедшие проверку пути или пользователя, отбрасываются. Формируется список "применимых" правил.

### 2. Побитовая проверка прав (Bit-by-Bit Evaluation)

Права (READ, WRITE, DELETE, CREATE) вычисляются **независимо** друг от друга. 

Для каждого бита `B`:
1. Из списка "применимых" фильтруются только те ("релевантные"), которые содержат бит `B` в своём маске `permissions`.
2. Если для бита `B` не нашлось ни одного правила, то ответ равен `!default_deny` (если deny стоит `True` по умолчанию, то бит запрещен, если `False` — разрешен).
3. Если релевантные правила есть, алгоритм находит **единственное правило-победитель** (Winner Rule), которое и определит исход операции.
4. Значение бита зависит от флага `deny` правила-победителя (если `deny=False`, то разрешено, если `True` — запрещено).

### 3. Определение правила-победителя (Tie-Breaker System)

Для того, чтобы найти самое значимое правило (Победитель) для данного бита, используется тройная сортировка — от наименее важного признака к наиболее важному. Последнее правило после сортировки становится победителем.

Формула "силы" (strength) правила состоит из:

**1. Паттерн-Специфичность (Specificity)** `[Lowest weight]`
Чем точнее описан путь файла, тем "сильнее" правило. Это решает коллизии, когда оба правила имеют одинаковый приоритет (например, общая политика и специфика в одной папке).
- Подсчёт специфичности:
    - За каждый слэш `/` +10 очков (чем глубже путь, тем он специфичнее).
    - За каждый wildcard `*` штраф -10 очков.
    - За двойной `**` штраф -20 очков.
    - Идеальные (exact) совпадения без wildcards получают бонус +1000 очков.
- Пример: `projects/secret/pass.md` (exact) "сильнее", чем `projects/secret/*.md`, которое в свою очередь "сильнее" чем `**/*`.

**2. Приоритет (Priority)** `[Medium weight]`
Явный параметр `priority: int` (по умолчанию 0), который задается в YAML. 
Правило с `priority: 50` всегда побьёт правило с `priority: 40`, независимо от того, насколько общий у него паттерн.

**3. Запрет (Deny Flag)** `[Highest weight in tie-breaking]`
При абсолютно равных специфичности и приоритете (когда два одинаково сильных правила конфликтуют), система придерживается принципа наименьших привилегий — **Deny wins**.
Если произошла коллизия, то `deny=True` перевешивает `allow`.

### 4. Результирующий набор

Вычислив "победителей" для каждого отдельного флага (`READ`, `WRITE`, `DELETE`, `CREATE`), движок конструирует итоговую маску `Permission`.
Фасад (`VaultGatekeeper`) использует эту маску для выдачи ответов по запросу `.can_read()`, применяя битовое "И" (`effective & Permission.READ == Permission.READ`).

## 5. Inline-правила из Frontmatter

Файлы могут содержать секцию `acl:` в YAML-заголовке (frontmatter). Такие правила:
- **Автоматически привязываются** к точному пути файла (pattern = путь файла, exact match, specificity = 1000+).
- **Получают высокий базовый приоритет** (`priority_base = 200` по умолчанию), что позволяет им перекрывать большинство глобальных правил из `policy.yaml`.
- Участвуют в **том же алгоритме резолвинга**, что и глобальные правила — проходят через побитовую проверку, сортировку по `(priority, specificity, deny_flag)` и выбор победителя.

Таким образом, автор файла может:
- Выдать персональный доступ конкретному пользователю, даже если глобальная политика запрещает.
- Заблокировать операцию (например, удаление) для определённой роли, даже если глобально она разрешена.

Порядок приоритетов (от слабого к сильному):
1. Глобальные правила `policy.yaml` (priority ~0–100)
2. Inline-правила из frontmatter (priority ~200+)
3. Глобальные правила с очень высоким priority (>200) — если администратор хочет гарантированно перекрыть inline-правила
